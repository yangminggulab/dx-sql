\begin{definition}[SELECT 语句]
在 SQL 中，\textbf{SELECT} 语句是用于从数据库表中检索数据的核心指令。其基本语法结构由以下两个核心部分组成：
\begin{itemize}
\item \textbf{选什么 (What to select)}：紧跟在 \texttt{SELECT} 关键字之后，用于指定需要查询的列名（字段）。
\item \textbf{从哪里选 (From where to select)}：紧跟在 \texttt{FROM} 关键字之后，用于指定数据所在的表名。
\end{itemize}
\end{definition}

\begin{note}
\textbf{GROUP BY 的使用注意事项：}
\begin{itemize}
    \item \texttt{GROUP BY} 子句中列出的每一项都必须是检索列或有效的表达式，但\textbf{不能是聚集函数}。也就是说，\texttt{GROUP BY} 后面应当写“具体的分组依据”，而不能写 \texttt{COUNT(*)}、\texttt{SUM(price)} 这类分组之后才得到的结果。

    \item 如果在 \texttt{SELECT} 中使用了表达式，例如 \texttt{YEAR(order\_date)}，那么在 \texttt{GROUP BY} 中也必须写相同的表达式，而不能只按原列分组。

    \item 在很多 SQL 实现中，\texttt{GROUP BY} 中不能直接使用 \texttt{SELECT} 子句里定义的别名，因此通常应写原表达式，而不要写别名。

    \item 大多数 SQL 实现不允许对长度可变的大字段进行分组，例如文本型字段或备注型字段。
\end{itemize}
\end{note}
\begin{note}
\textbf{WHERE 与 HAVING 的区别：}
\begin{itemize}
    \item \texttt{WHERE} 用于在分组之前对原始记录进行筛选，决定哪些行能够进入分组，因此 \texttt{WHERE} 是“管行”的。
    \item \texttt{HAVING} 用于在分组之后对分组结果进行筛选，决定哪些分组能够保留下来，因此 \texttt{HAVING} 是“管组”的。
    \item 一般来说，\texttt{WHERE} 中不能使用聚集函数，而 \texttt{HAVING} 中可以使用聚集函数。
\end{itemize}
\end{note}

\begin{example}
下面的 SQL 语句先筛选工资大于 5000 的员工，再按部门编号分组，最后只保留人数不少于 3 的部门：
\begin{lstlisting}[language=SQL]
SELECT dept_id, COUNT(*)
FROM employee
WHERE salary > 5000
GROUP BY dept_id
HAVING COUNT(*) >= 3;
\end{lstlisting}
\end{example}

\begin{note}
上述语句的执行过程可以理解为：
\begin{itemize}
    \item 先由 \texttt{WHERE salary > 5000} 筛选出工资大于 5000 的员工记录；
    \item 再按照 \texttt{dept\_id} 对这些记录进行分组；
    \item 最后由 \texttt{HAVING COUNT(*) >= 3} 保留员工人数不少于 3 的那些部门分组。
\end{itemize}
\end{note}
\begin{dxtips}
    聚集函数如果和 GROUP BY 一起出现，默认就是“对每个组聚集”。所以聚类函数(aggregate function)count all默认就是count刚分好的组。
\end{dxtips}
\begin{note}
虽然 \texttt{SELECT} 子句写在最前面，但 SQL 查询在逻辑上并不是先执行 \texttt{SELECT}。一般可理解为按如下顺序执行：
\[
\texttt{FROM} \;\to\; \texttt{WHERE} \;\to\; \texttt{GROUP BY} \;\to\; \texttt{HAVING} \;\to\; \texttt{SELECT} \;\to\; \texttt{ORDER BY}.
\]
因此，\texttt{COUNT(*)} 虽然写在 \texttt{SELECT} 中，但如果语句里有 \texttt{GROUP BY}，它实际统计的是每个分组中的行数。
\end{note}

\begin{example}
下面这个写法是一个典型错误：
\begin{lstlisting}[language=SQL]
SELECT cust_email
FROM Customers
WHERE cust_id IN (
    SELECT order_date, cust_id
    FROM Orders
    WHERE order_num IN (
        SELECT order_num
        FROM OrderItems
        WHERE prod_id = 'BR01'
    )
);
\end{lstlisting}

错误原因在于：外层条件是
\[
\texttt{cust\_id IN (子查询)}
\]
这里左边只有一个值 \(\texttt{cust\_id}\)，因此括号中的子查询必须只返回一列，并且这一列应当能够与 \(\texttt{cust\_id}\) 进行比较。

但是该子查询返回了两列：
\[
\texttt{order\_date,\ cust\_id}
\]
这就与 \texttt{IN} 的要求不匹配，因此会报错。

正确写法应为：
\begin{lstlisting}[language=SQL]
SELECT cust_email
FROM Customers
WHERE cust_id IN (
    SELECT cust_id
    FROM Orders
    WHERE order_num IN (
        SELECT order_num
        FROM OrderItems
        WHERE prod_id = 'BR01'
    )
);
\end{lstlisting}

也就是说，当写成
\begin{lstlisting}[language=SQL]
某列 IN (子查询)
\end{lstlisting}
时，子查询通常只能返回一列；如果返回多列，就会出现这种典型错误。
\end{example}
\begin{note}
    SQL 里子查询常见有三种位置：
	1.	写在 WHERE 后面：拿来筛选
	2.	写在 FROM 后面：拿来当一张临时表
	3.	写在 SELECT 后面：拿来给每一行补一个值
    select后面可以是表中没有的也就是处理过的新添加的
\end{note}
\begin{example}
下面的查询用于返回每个顾客的顾客 ID 以及其已订购商品的总金额，并按总金额从大到小排序：
\begin{lstlisting}[language=SQL]
SELECT cust_id,
       (SELECT SUM(quantity * item_price)
        FROM OrderItems
        WHERE order_num IN (SELECT order_num
                            FROM Orders
                            WHERE Orders.cust_id = Customers.cust_id)) AS total_ordered
FROM Customers
ORDER BY total_ordered DESC;
\end{lstlisting}
\begin{dxtips}
    先把最终要输出的 \verb|cust_id| 和 \verb|as total_orders| 写在 \verb|select| 后，这里子句明显是作为计算字段，因为要添加表里面原来没有的东西。所以 \verb|select| 肯定在 \verb|from| 前面。一共需要两步，我们先算总价格，这个是容易想的；但是算总价格这步不能建立起 \verb|orders| 和 \verb|customers| 的相同 id 的映射，因为 \verb|where in| 这种筛选只能里面有一个 \verb|select|，所以还需要一个筛选的 \verb|select|。最后有一个 \verb|as| 的名字就行了，\verb|sum| 的时候就不用有名字。
\end{dxtips}

\begin{dxtips}[一个顾客可以下好几个订单，每一个订单可以买好几个东西]
    建立两个表的联系就是 \verb|where A in (select A from ...)|，\(A\) 的位置只能是一个。这里如果要计算 \verb|sum|，就需要知道 \verb|sum| 的是那个名字的所有的数量 \(*\) 价格，所以我们必须要筛选相同 \verb|order_num| 的东西。写出标题话我们就可以知道，算订单总价紧跟着的必须是订单号，筛选订单号紧跟着的必须是顾客 id。可以从侧栏点当前输入，还可以检查这个表里面到底有没有。
\end{dxtips}

此语句中的 \verb|SELECT| 与前面的 \verb|SELECT| 语句相同，但 \verb|FROM| 子句不同。这里，两个表之间的关系是以 \verb|INNER JOIN| 指定的部分 \verb|FROM| 子句。在使用这种语法时，联结条件用特定的 \verb|ON| 子句而不是 \verb|WHERE| 子句给出。传递给 \verb|ON| 的实际条件与传递给 \verb|WHERE| 的相同。
\end{note}

\begin{example}
在多表联结中，若某个字段名在多个表中都出现过，就需要写成“表名.\allowbreak 字段名”的形式，以消除歧义。例如这里的 \verb|order_num| 在 \verb|Orders| 表和 \verb|OrderItems| 表中都可能出现，因此应写成 \verb|Orders.order_num|。

另外，若在 \verb|SELECT| 中使用了聚合函数 \verb|sum(...)|，那么凡是同时出现但没有被聚合的字段，都必须写在 \verb|GROUP BY| 中。所以 \verb|cust_name| 和 \verb|Orders.order_num| 都要分组。

对应的 SQL 语句为：
\begin{lstlisting}[language=SQL]
select cust_name, Orders.order_num,
       sum(quantity*item_price) as ordertotal
from Customers
inner join Orders
on Customers.cust_id = Orders.cust_id
inner join OrderItems
on OrderItems.order_num = Orders.order_num
group by cust_name, Orders.order_num;
\end{lstlisting}
\end{example}
\begin{definition}[等联结]
若两个表进行联结时，联结条件是两个字段之间的相等关系，即使用等号 \(=\) 来建立匹配关系，则这种联结称为\textbf{等联结}。

例如：
$
\texttt{Customers.cust\_id = Orders.cust\_id}
$
和
$
\texttt{Orders.order\_num = OrderItems.order\_num}
$
都是等联结条件。
\end{definition}

\end{dxtips}
\begin{example}
下面语句用于查询订单总额不少于 1000 的顾客编号和订单编号：
\begin{lstlisting}[language=SQL]
select Orders.cust_id, OrderItems.order_num
from Orders
inner join OrderItems
on Orders.order_num = OrderItems.order_num
group by Orders.cust_id, OrderItems.order_num
having sum(item_price * quantity) >= 1000;
\end{lstlisting}

常见错误如下：
\begin{enumerate}
    \item 将 \texttt{OrderItems} 误写为 \texttt{OrdersItems}。
    \item 将聚合条件写在 \texttt{where} 中，而不是写在 \texttt{having} 中。
    \item \texttt{select} 中出现的非聚合列没有全部写入 \texttt{group by}。
    \item 多表查询时，对重名字段没有写成 \texttt{表名.字段名}，从而产生歧义。
\end{enumerate}
\end{example}
\begin{lstlisting}[language=SQL]
SELECT cust_name, cust_contact
FROM Customers AS C, Orders AS O, OrderItems AS OI
WHERE C.cust_id = O.cust_id
  AND OI.order_num = O.order_num
  AND prod_id = 'RGAN01';
\end{lstlisting}
这样就可以缩短表头的书写。
\begin{note}[自联结]
    可以，因为这里不是两个不同的 \texttt{Customers} 表，而是同一张表起了两个别名：
\begin{itemize}

    \item \texttt{Customers AS c2}
\end{itemize}

相当于把同一张表“当成两份来看”。

这样做是为了让表自己和自己比较，这种连接称为\textbf{自联结}（self join）。

这句查询语句的含义是：
\begin{itemize}
    \item \texttt{c2.cust\_contact = 'Jim Jones'}：先找到联系人是 \texttt{Jim Jones} 的那一行；
    \item 再利用 \texttt{c1.cust\_name = c2.cust\_name}，找到与他姓名相同的其他顾客。
\end{itemize}

所以，\texttt{c1} 和 \texttt{c2} 只是同一张表在查询中扮演的两个不同角色，并不冲突。
\end{note}

\begin{example}
在 SQL 中，\texttt{UNION} 要求两个 \texttt{SELECT} 的结果结构兼容，即列数相同，且对应位置的数据类型可以匹配；它并不要求两边查询的是同一种含义的数据。

例如下面是可以的：
\begin{lstlisting}[language=SQL]
SELECT prod_name
FROM Products
UNION
SELECT cust_name
FROM Customers;
\end{lstlisting}

因为两边都只返回 1 列，且通常都是字符型。这个的效果就是两个合成一列

但下面不可以：
\begin{lstlisting}[language=SQL]
SELECT prod_name, prod_id
FROM Products
UNION
SELECT cust_name
FROM Customers;
\end{lstlisting}

因为左边返回 2 列，而右边只返回 1 列，列数不一致。
\end{example}

\begin{example}[JOIN 视图]
创建仅含下过单顾客的视图：
\begin{lstlisting}[language=SQL]
CREATE VIEW CustomersWithOrders AS
SELECT Customers.*, Orders.order_num, Orders.order_date
FROM Customers
JOIN Orders ON Customers.cust_id = Orders.cust_id;
\end{lstlisting}
常见错误：\texttt{JOIN} 后漏写 \texttt{ON}；用 \texttt{SELECT *} 导致 \texttt{cust\_id} 重复（Error 1060）；重建视图前未执行 \texttt{DROP VIEW}（Error 1050）。注意 \texttt{INNER JOIN} 已自动过滤未下单顾客，无需额外 \texttt{WHERE}。
\end{example}

\end{example}
\section{高级用法}
\begin{definition}[存储过程]
存储过程（Stored Procedure）是一组预编译的SQL语句集合，存储在数据库中可反复调用，语法为：
\begin{lstlisting}[language=SQL, frame=single, backgroundcolor=\color{white}]
CREATE PROCEDURE 过程名(参数)
BEGIN
    -- 过程体
END;

-- 调用
CALL 过程名(参数);
\end{lstlisting}
与函数的区别在于：存储过程无需返回值，不能嵌套在 \texttt{SELECT} 中使用。
\end{definition}

\begin{definition}[事务处理]
事务处理（Transaction）是一组SQL操作的集合，要么全部执行成功提交，要么全部回滚撤销，保证数据的一致性。核心语句为：
\begin{lstlisting}[language=SQL, frame=single, backgroundcolor=\color{white}]
START TRANSACTION;  -- 开始事务
COMMIT;             -- 提交（全部成功则执行）
ROLLBACK;           -- 回滚（出错则撤销所有操作）
\end{lstlisting}
事务具有四个基本性质（ACID）：原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）、持久性（Durability）。
\end{definition}

\begin{definition}[游标]
游标（Cursor）是一种数据库对象，用于逐行处理查询结果集，语法为：
\begin{lstlisting}[language=SQL, frame=single, backgroundcolor=\color{white}]
DECLARE 游标名 CURSOR FOR SELECT语句;
OPEN 游标名;
FETCH 游标名 INTO 变量名;
CLOSE 游标名;
\end{lstlisting}
游标通常在存储过程中使用，适用于需要对结果集逐行操作的场景。
\end{definition}

\begin{definition}[约束]
约束（Constraint）是对表中数据的限制规则，常见类型如下：
\begin{lstlisting}[language=SQL, frame=single, backgroundcolor=\color{white}]
PRIMARY KEY,          -- 主键，唯一标识每一行
FOREIGN KEY,          -- 外键，保证引用完整性
UNIQUE,               -- 唯一约束，列值不可重复
NOT NULL,             -- 非空约束
CHECK (条件)          -- 检查约束，限制列的取值范围
\end{lstlisting}
\end{definition}

\begin{definition}[索引]
索引（Index）是对表中一列或多列的值建立的排序结构，用于加快查询速度，语法为：
\begin{lstlisting}[language=SQL, frame=single, backgroundcolor=\color{white}]
CREATE INDEX idx_name ON 表名 (列名);
DROP INDEX idx_name ON 表名;
\end{lstlisting}
索引可以提升查询效率，但会降低插入、更新、删除的速度，且占用额外存储空间。
\end{definition}

\begin{definition}[触发器]
触发器（Trigger）是在特定事件（\texttt{INSERT}、\texttt{UPDATE}、\texttt{DELETE}）发生时自动执行的SQL语句，语法为：
\begin{lstlisting}[language=SQL, frame=single, backgroundcolor=\color{white}]
CREATE TRIGGER 触发器名
AFTER INSERT ON 表名
FOR EACH ROW
BEGIN
    -- 触发时执行的操作
END;
\end{lstlisting}
触发器常用于自动记录日志、维护数据一致性等场景。
\end{definition}
